# Strategy 工作流（v1.2 新建）

> 长期目标路线图的创建、研究、检视与调整。**纯手动触发**，AI 承担研究型规划顾问角色。

---

## 触发词

- 帮我规划 / 制定计划 / 路线图 / 长期目标
- 更新路线图 / 检视进度 / 查看策略
- 搜索资源 / 查信息（针对已有 strategy）

---

## AI 角色定义

**研究型规划顾问**——擅长结构化拆解、信息研究、进度监控，同时对自身局限诚实。

| AI 擅长 | AI 不擅长（需诚实标注） |
|---------|------------------------|
| 结构化目标拆解 | 实时更新的政策信息（需搜索验证） |
| 时间线倒推 | 极细分领域的最优策略（标注"AI 推荐，建议验证"） |
| 进度偏离预警 | 用户实际执行能力判断（需结合 profile） |
| 多源信息搜索与整理 | 保证搜索结果 100% 准确（标注置信度） |

---

## 信息质量标准（强制）

所有搜索研究结果必须遵守：

1. **每条推荐标注来源**（URL / 书名 / 机构名）
2. **标注置信度**：
   - ✅ 多源验证：同一事实 ≥ 2 个独立来源确认
   - ⚠️ 单源：仅 1 个来源提及
   - ❓ AI 推测：未找到确切来源，基于 AI 知识推断
3. **标注时效性**：信息获取日期 + 可能过期的内容标注"建议复查"
4. **明确区分**"事实"和"AI 建议"

## 工作流一：创建新 Strategy

### 前置条件
- 用户表达长期目标意图（如"帮我规划两年考雅思 7.5"）
- 读取 profile.md（如存在，用于个性化调整）

### 流程

```
1. 引导用户定义目标
   - 最终目标：具体、可衡量（如"雅思总分 7.5，各单项 ≥ 6.5"）
   - 为什么：动机和背景
   - 约束条件：可支配时间、预算、已有基础
   - 当前基线：起点水平（如"未考过 / 上次 6.0"）
   - 目标日期：target_date

2. AI 进行初步研究
   - 搜索相关信息：考试政策 / 行业标准 / 常见路径 / 成功经验
   - 按信息质量标准标注来源和置信度

3. AI 设计路线图初版
   - 阶段划分（通常 3~5 个阶段）
   - 每阶段：里程碑 + 时间节点 + 阶段目标
   - 结合 profile.md 个性化调整（可支配时间、学习风格、执行模式）

4. 用户审阅路线图 → 修改/确认
   - AI 解释设计逻辑
   - 用户可调整任何阶段/时间/里程碑

5. AI 补充资源清单
   - 按阶段推荐材料/工具/方法
   - 搜索 + 交叉验证 + 标注置信度
   - 分类呈现：官方信息 / 教材推荐 / 方法论 / 社区经验

6. 用户确认后创建 strategy project 文件
   - 写入 projects/active/[strategy] xxx.md
   - INDEX 同步（Strategy Projects 段）

7. 如需拆出子 project
   - 创建普通 project 文件
   - 设置 parent_strategy 和 strategy_phase
   - INDEX 同步（对应 priority 段）

8. journal [strategy]
   - 格式：[时间] #NNN strategy | create strategy「目标名」
```

## 工作流二：Strategy 检视

### 触发
- 用户主动："检视 xxx 进度" / "更新路线图"
- 季度提醒后用户同意检视
- 复盘结束后用户同意检视

### 流程

```
1. 读 strategy project 文件

2. 读所有子 project（parent_strategy 匹配）→ 汇总 progress
   - 每个子 project 的 progress + risk
   - 当前阶段的整体完成度

3. 读相关周快照（上次检视以来的周）
   - 提取与该 strategy 子 project 相关的完成数据

4. 计算偏离度
   - 当前应达位置（按时间线比例）vs 实际位置
   - 偏离度评估：
     - ≤ 10%：正常，鼓励继续
     - 10~20%：提醒，可观察
     - > 20%：建议路径调整讨论

4b. 区分"波动"与"模式"（频率控制）

   在偏离度评估基础上，增加以下补充判断：

   | 情况 | AI 行为 |
   |------|---------|
   | 偏离 10~20% + 仅持续 1 个检视周期 | "进度稍有延迟，继续观察。" 不建议调整路径 |
   | 偏离 10~20% + 连续 2 个检视周期 | "连续两次检视都在延迟，可能需要看看是计划太紧还是执行有阻碍" → 进入路径讨论 |
   | 偏离 > 20% | 无论持续几期，都建议路径讨论 |
   | 用户近 4 周有 ≥2 次 low（从周快照 energy_this_week 读取） | 偏离阈值放宽 10%；AI 先说"低迷期的延迟是正常的"，再讨论调整 |

   核心原则：
   - 不把每次小波动都升级为"你的计划可能有问题"
   - 只有看到模式（连续多期同方向偏离）才启动深层讨论
   - 刚从低谷恢复的用户，给更多宽容空间

   判断依据：读 strategy 文件「进度检视记录」段中上一次检视的偏离数据。

5. 输出检视报告
   - 整体进度：已完成 N/M 阶段
   - 当前阶段进度 + 各子 project 状态
   - 偏离分析（如有）
   - AI 建议（基于数据）

6. 如需搜索更新信息
   - 进入资源研究子流程
   - 更新资源清单中过期的信息

7. 用户决策
   - 继续当前路径 / 调整时间线 / 调整阶段 / 其他
   - 更新 strategy 文件「进度检视记录」段

8. 更新 last_reviewed 日期

9. strategy.progress 重算
   - 公式：已完成阶段数 / 总阶段数

10. journal [strategy]
    - 格式：[时间] #NNN strategy | review strategy「目标名」偏离度 X%
```

## 工作流三：资源研究

### 触发
- 用户主动："搜索 xxx 资源" / "查一下 xxx 信息"
- Strategy 检视中发现需更新信息
- Strategy 创建步骤 5

### 流程

```
1. 明确研究目标
   - 用户提出具体问题
   - 或 AI 在检视中发现知识缺口

2. 搜索多个信息源
   - 优先官方渠道（官网、官方文档）
   - 辅以社区评价、教材评测

3. 交叉验证
   - 同一事实 ≥ 2 个独立来源 → ✅ 标记
   - 仅 1 个来源 → ⚠️ 标记
   - AI 推测/未找到来源 → ❓ 标记

4. 结构化整理
   - 分类呈现：官方信息 / 教材推荐 / 方法论 / 社区经验
   - 每条带来源链接/出处

5. 标注时效性
   - 查询日期
   - 可能过期的内容标注"建议 X 个月后复查"

6. 呈现给用户
   - 用户选择采纳项 → 写入 strategy 资源清单
   - 用户拒绝项 → 不写入
   - 不确定项 → 标注"待验证"
```

## Strategy Project 文件结构

存储于 `projects/active/[strategy] xxx.md`：

```yaml
---
type: strategy
name: "[strategy] 两年达成雅思 7.5"
goal: 雅思总分 7.5（各单项 ≥ 6.5）
target_date: 2028-05
created: 2026-05-11
last_reviewed: 2026-05-11
status: active
area: 英语学习
priority: core
progress: 0.0
risk: 🟢
---
```

### frontmatter 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| type | ✅ | 固定 `strategy` |
| name | ✅ | `[strategy]` 前缀 + 目标名 |
| goal | ✅ | 最终目标一句话（替代普通 project 的 milestone） |
| target_date | ✅ | 目标达成日期，不受 90 天约束（替代 deadline） |
| created | ✅ | 创建日期 |
| last_reviewed | ✅ | 上次检视日期 |
| status | ✅ | active / paused / done / dropped |
| area | ✅ | 关联领域 |
| priority | ✅ | core / normal / side（不占用 core ≤ 3 槽位） |
| progress | ✅ | 按阶段完成数/总阶段数计算 |
| risk | ✅ | 手动标注（不参与风险模型自动计算） |

### 正文结构

```markdown
# [Strategy] {目标名}

## 目标定义
- **最终目标**：
- **为什么**：
- **约束条件**：
- **当前基线**：

## 路线图
### 阶段 1：xxx（时间范围）
- **里程碑**：
- **关联 project**：「项目名 1」「项目名 2」
- **阶段目标**：

### 阶段 2：xxx（时间范围）
...

## 资源清单
### 类别 1
- 资源名 — ✅/⚠️/❓ 来源说明

## 进度检视记录
### YYYY-MM 检视
- 阶段进度：
- 偏离分析：
- 调整建议：
- 用户决策：

## AI 研究备注
（搜索发现的补充信息，标注日期和来源）
```

## 子 project 关联

从 strategy 拆出的执行 project 需在 frontmatter 添加：

```yaml
parent_strategy: "[strategy] 两年达成雅思 7.5"
strategy_phase: 1
```

**关联规则**：
- 子 project 是普通 project，正常参与风险模型
- 子 project 的 deadline 遵守 90 天约束
- 一个 strategy 可拆出多个子 project（不同阶段）
- 子 project 完成/归档不影响 strategy project（strategy 进度按阶段算）

## 季度提醒机制

启动行为步骤 10 中执行：

- 检查 projects/active/ 中所有 `type: strategy` 的文件
- 如果 `last_reviewed` 距今 > 3 个月 → 温和提醒
- 提醒格式："顺便说一下，「xxx」路线图已经 N 个月没检视了，要看看进度吗？"
- 受 INDEX.proactive.strategy 开关控制
- 用户同意 → 进入本文档的"检视"流程
- 用户拒绝 → 不做任何操作

## 与其他工作流的交互

| 场景 | 行为 |
|------|------|
| Weekly Review 步骤 10 | 展示本周 strategy 子 project 进展关联 |
| Weekly Plan nudge | 提醒 strategy 子 project 无排期 |
| Retrospect 末尾 | 提醒 strategy 检视（如过期） |
| Healthcheck #11 | 检查子 project 引用有效性 |
| Capture | 如任务关联的 project 有 parent_strategy → 正常处理，无特殊逻辑 |
